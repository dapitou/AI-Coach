# 📄 产品需求文档 (PRD)：K1力量镜 - 电机异常交互系统

## 一、 设计理念与全局原则

1.  **非全局阻断**：任何电机故障**绝不**锁死整个App。电机坏了，用户依然可以浏览课程、查看数据、甚至投屏看视频，仅在进入“需要调动电机”的训练页面时进行拦截。
2.  **情绪安抚**：文案避免使用“致命错误”、“严重损坏”等惊吓词汇，改用“小憩一下”、“需要您的帮助”、“设备遇到小状况”等安抚性语言。
3.  **分层引流**：能通过“等温降”、“拔插头”解决的，坚决不弹售后；只有确诊硬件死亡的，提供一键专属售后通道，降低客服成本。
4.  **全景遥测**：无论故障大小，后台静默打包故障发生前10秒的物理快照（拉力、速度、温度、电压等），为硬件和算法团队提供完美的“案发现场”。

## 二、 故障分级与交互策略定义

我们将16个错误码抽象为四大体验场景（Scene A/B/C/D）：

### 🔴 硬件阻断级（确诊物理损坏）
*   **命中错误码**：`1, 2` (硬件过流)、`6, 7` (相电流零偏)、`8, 9` (缺相保护)。
*   **产品逻辑**：硬件已判定不可逆异常，继续通电有安全隐患。
*   **交互表现**：
    *   **强打断**：立即强制中断当前训练，电机失能（泄力）。
    *   **UI呈现**：弹出【居中模态弹窗】，用户点击“我知道了”后，弹窗消失，但在App全局顶部常驻【红色告警Banner】（点击Banner可再次呼出弹窗）。
    *   **阻断逻辑**：拦截一切“使能/设力”操作，点击开始训练时提示“设备需检修，无法启动”。
    *   **安抚文案**：“设备动力系统遇到了一点小状况，为了您的安全，已进入保护模式。请不用担心，点击下方按钮，我们的专属工程师将为您处理。”
    *   **行动点**：[一键上传日志并联系售后] | [返回首页]

### 🟠 环境与疑难级（需用户自助排查）
*   **命中错误码**：`5` (相电流过流 / 断绳保护 ⚠️复合错误码)。
*   **产品逻辑**：由于底层未区分过流与断绳，必须通过App引导用户做“两步走”排查。
*   **交互表现**：
    *   **强打断**：立即中断训练，电机失能。
    *   **UI呈现**：弹出【引导排查弹窗】，采用图文/动图形式引导。
    *   **安抚文案**：“哎呀，动力输出被打断了。可能是刚刚力量太猛，或者是拉绳遇到了问题。”
    *   **自助行动流**：
        1.  *检查拉绳*：“请先检查两侧拉绳是否完好。若已断裂，请点击 [登记换绳]。”
        2.  *重启设备*：“若拉绳完好，请尝试拔下设备电源插头，静置30秒后重新插上。”
        3.  *兜底*：“如果重启后仍然出现此提示，请 [联系售后]。”

### 🟡 疲劳与降级级（温度/电压波动）
*   **命中错误码**：`3, 4, 16` (NTC传感器失效)、`10, 11, 12` (过温)、`13` (电压异常)。
*   **产品逻辑**：设备“练累了”或外界电网波动，只需等待即可恢复。底层已做最高10kg的降力保护或停机。
*   **交互表现**：
    *   **弱打断/软限制**：不退出训练页面，但UI上的重量调节盘最高被**锁死在10kg**（置灰超限部分）。
    *   **UI呈现**：训练页顶部下发【黄色警告Banner】或【Toast提示】。
    *   **安抚文案**：
        *   *(针对过温/NTC)*：“设备刚刚经过了高强度训练，正在大口喘气散热中~ 当前最大助力已限制为10kg，建议您也拉伸休息片刻哦。待设备冷却后将自动恢复。”
        *   *(针对电压)*：“检测到您家中的市电电压有波动，设备正在保护性暂停，待电压平稳后即可继续。”
    *   **恢复逻辑**：后台轮询温度/电压状态，一旦底层上报故障清除，立刻恢复重量上限，并Toast提示：“设备已满血复活，继续挑战吧！”

### 🔵 运动安全保护级（脱手飞车）
*   **命中错误码**：`14, 15` (脱手保护)。
*   **产品逻辑**：极度危险的运动场景。底层会在1秒内急停并重置为基础力。App必须让用户明确感知到“力量没了”，防止二次受伤。
*   **交互表现**：
    *   **语音强提示**：伴随警报音效，立刻播报语音：“检测到脱手，力量已重置，请注意安全！”
    *   **数据打断**：立即终止当前的AI计次循环，强制标记当前动作为“异常/作废”。
    *   **UI呈现**：屏幕中央闪烁红色警告动效，提示“脱手保护已触发，需重新设定力量”。
    *   **恢复逻辑**：强制要求用户在App上重新点击“开始训练”或重新调节重量后，才下发新的使能指令（不采用底层静默恢复，确保用户有心理预期）。

## 三、 遥测与日志上报策略

为了帮助硬件、算法研发团队彻底排查问题，上位机必须建立强大的**“异常快照 (Crash Snapshot)”**机制。

当任何 >= 1 的故障码触发时，App 层需在后台静默执行以下操作：

1.  **触发快照收集器**：
    *   捕获 `timestamp` (触发时间)。
    *   捕获当前 App 的 `Scene/Activity` (在做什么训练，哪个动作)。
    *   打包最近 **10秒** 的电机核心数据流（以10Hz频率）：`左/右拉力(kg)`、`左/右速度(cm/s)`、`左/右位置(cm)`。
    *   抓取当前的硬件状态切片：`电机0/1温度`、`驱动板温度`、`母线电压`。
2.  **上报与归档机制**：
    *   将上述快照生成 JSON 文件，命名格式：`K1_Error[故障码]_[用户ID]_[时间戳].json`。
    *   后台尝试静默上传至自研 OSS / AWS S3。若网络断开，则缓存在本地 `/logs/motor_faults` 目录下，待网络恢复后重传。
    *   对于用户主动点击“一键联系售后”的操作，将此 JSON 的 OSS URL 附加在售后工单的隐藏字段中。

## 四、 PM 核心跟进事项

为了让这套体验完美落地，我们需要在接下来的冲刺中协调以下资源：

1.  **UI/UX 团队**：输出 A/B/C/D 四种场景的 UI 视觉稿。特别是 Scene B 的“排查引导图”和全局的“降级状态Banner”。
2.  **固件/硬件团队**：再次强调，错误码 `5` 的揉面设计（过流和断绳放一起）是最大的体验隐患。由于当前已无法修改，需硬件团队给出**辅助判定算法**（例如：发生错误5时，若前2秒的 `[side]_velocity` 超过 300cm/s，App 层大概率判定为断绳；否则判定为过流），由我们在安卓端做“软区分”。
3.  **QA 测试团队**：在下个测试用例中，需强行注入模拟故障码，验证 App 训练页面在收到故障码后，视频课程是否仍能流畅播放不 Crash。
